## 二、Phase 依賴關係

```
P1（基礎設施）
  └─ P2（專案與成員）
       └─ P3（資料管理）
            └─ P4（會議基礎）
                 ├─ P5（Bot Session）
                 │    └─ P6（摘要）
                 └─ P7（前端，可與 P4+ 並行啟動）
```

P4 → P5 耦合緊密，建議同一個聊天室完成。  
P7 前端可在 P4 完成後獨立開展，後端 API 已定義完整不需等待 P5/P6。

## 四、聊天室策略建議

每個 Phase 使用**獨立聊天室**。進入新聊天室前需附帶：

1. `CLAUDE.md`（根目錄，P1 建立後每 Phase 更新）
2. 當前 Phase 的計畫文件（`09-實作計畫/0N-PhaseN-*.md`）
3. 相關設計文件（依 Phase 列在各 Phase 文件的「參考文件」欄）
4. 前一個 Phase 已建立的目錄結構清單（`tree` 輸出即可）

**例外**：P4 和 P5 耦合緊密（session 建立在 meeting.service 呼叫後觸發），可合併為同一聊天室。

## 五、各 Phase 的 Smoke Test 原則

每個 Phase 完成後，在進入下一個 Phase 前執行：

- **P1**：`npx ts-node src/index.ts` 啟動不拋錯；`GET /me` 能正確回傳 401
- **P2**：`POST /projects` 正常建立，Dify dataset 在 Dify 控制台可見
- **P3**：上傳 PDF 後 Material 出現在 Dify Knowledge Base；刪除後三方均同步消失
- **P4**：`POST /projects/:id/meetings` 後 Vexa 控制台顯示 Bot 正在加入對應 Meet
- **P5**：在 Google Meet 說「蜜塔，現在幾點？」Bot 以語音回覆
- **P6**：Bot 離會後 5~30 秒內，Meeting Detail 頁出現摘要
- **P7**：完整走完「建立專案 → 上傳檔案 → 建立會議 → 問答 → 結束會議」流程
