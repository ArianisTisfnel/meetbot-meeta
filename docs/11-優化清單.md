## 釐清紀錄（Q&A）

> 「想確認」類問題的調查結論，作為後續決策依據。

### 📘 U10 — 會議何時結束？如何把蜜塔請出去？
**會議只在以下三種情況轉為結束（ENDED/FAILED）**（`sessions/session-manager.ts`）：
1. **使用者主動結束**：會議詳情頁的「結束會議」按鈕 → `leaveMeeting` → 對 Vexa 發 `DELETE /bots` 撤回蜜塔 → `handleSessionClose`。
2. **Vexa 送終止事件**：WS 收到 `completed`/`failed`/`needs_help`/`needs_human_help`（蜜塔自己離開或 Vexa 判定結束）。
3. **服務重啟還原時**：發現 Vexa 端該會議已非 active。

**所以「只剩蜜塔一人不會結束」是正確現象**——只要蜜塔還在、Vexa 沒送終止事件，狀態就維持進行中。
**怎麼請它出去**：到該會議的**詳情頁**點右上角「結束會議」（需 `canMeeting` 權限）。
> ✅ 已實作（U10）：會議列表的進行中（ACTIVE）列已直接加「結束」快捷鍵，不必再進詳情頁。

### 📘 U11 — 摘要與其他業務流程何時觸發？
在 `handleSessionClose` 中，當會議轉為 **ENDED**（非 FAILED）時呼叫 `generateSummaryAsync`，依序執行：抓取完整逐字稿 → 產生摘要/行動項目（Dify/Claude）→ 逐字稿存至 Supabase Storage。**即「會議結束的當下」觸發**。FAILED 不會生成摘要。摘要 sentinel：`null`=生成中（前端持續輪詢）、`''`=已嘗試但無內容、有字串=完成。

### 📘 U14 — OAuth 能拿到哪些使用者資料？
Google OAuth 的 `profile` 可提供：**email、name、picture（頭像）、given_name、family_name、locale、sub（Google 唯一 ID）**。
目前我們**只用了 `profile.email`**（見 `lib/auth.ts`），name/picture 都被丟棄。但 NextAuth 預設會把 name/picture 放進 `session.user.name` / `session.user.image` → **前端其實已可直接顯示登入者的名稱與頭像**（U15 因此好做）。
注意：**成員列表顯示的是「其他使用者」的名稱**，來源是 Vexa `public.users.name`（目前為 null，見 U16），無法靠登入者自己的 session 補；要正確顯示所有成員名稱，需在建立 Vexa 使用者時把 name 一併寫入（或自建使用者資料表）。

### 📘 U16 — 帳號架構：Google 帳號還是自有帳號？
**目前是「Google OAuth 登入 + Vexa 使用者」的混合**：
- 登入用 **Google OAuth**（NextAuth GoogleProvider）。
- 首次登入時，後端用 email 呼叫 Vexa Admin API `POST /admin/users {email}` **建立/取得一個以 email 為鍵的 Vexa 使用者**，並發 API token。
- **我們沒有自己的帳號資料表**，完全以 Vexa 的 `public.users` 為準（`vexaUserId` 即 Vexa 使用者 id）。
- 🔑 **U8 根因確認**：建立 Vexa 使用者時**只送了 email、沒送 name**（`lib/auth.ts` 第 34 行），所以 `public.users.name` 永遠是 null。
- **改善路線**（待你決定，影響 U8/U14/U15）：
  - A. **沿用 Vexa**：建立使用者時把 Google 的 name 一起送進 Vexa（若 Admin API 支援 name 欄位），最省事、成員名稱即正確。
  - B. **自建使用者資料表**（app schema）存 name/avatar/加入時間等：彈性高、可做帳號面板，但要自行維護同步、與 Vexa 雙寫，複雜度上升。

## 二、進一步優化方向（已盤點，待排期）

> 這些是實作過程中發現、但本次未動的改善點，依優先序排列。

   → 建議：併計 PENDING。
3. **`leaveMeeting` 邊界**：session 已不在記憶體時 DB 不會轉 ENDED，但 API 仍回 ENDED，狀態不一致。
   → 建議：以 DB 結果為準回傳。
4. **`MEETING_END` 未入活動紀錄**：`handleSessionClose` 沒有 projectId，會議結束事件目前未記錄。
   → 建議：補查 projectId 後寫入，使活動紀錄更完整。
5. **清理 dead code**：`materialService.listHistory`、`components/ui/select.tsx`（無功能空殼）已無用。
   → 建議：移除。
6. **隱私選項**：成員 email 直接顯示。若有隱私顧慮，可加「隱藏 email」設定。

Bot 在候客室等待 15 分鐘後 timeout。會議 Host 完全看不到蜜塔的進入請求，但同一會議裡其他 Google 帳號的加入請求可以正常顯示。從無痕視窗以 Guest 身份（不登入 Google）手動加入同一個 Google Meet，Host 也看不到該請求 → 確認該 Google 帳號的 Meet 設定對非 Google 帳號的 Guest 請求有特殊處理。

**案例 B：Meeting #8（stopped_before_admission，Chrome 未啟動）**
Bot 進程啟動後 24 秒即崩潰，無任何 bot_logs（Chrome 未能啟動），Vexa 分類為 transient_infra。在此期間容器本身自動重啟過一次。

**案例 C：Meeting #12（failure_stage: joining，manual_leave）**
此次嘗試前，Host 在會議中途手動開啟「快速存取（Quick Access）」。Bot 啟動正常，Chrome 導航至 Google Meet，找到名字輸入欄位，但：

Bot 找到名字欄位後無後續動作，1 分鐘後自行離開。Vexa log 中無來自我們後端的 DELETE /bots 呼叫，確認非後端主動撤回。

**Ray 的成功案例（對照組）**
同一套 Vexa-lite 容器，user_id=1（raytsai615@gmail.com），5 月 17 日及 5 月 20 日各有一次成功紀錄：
- Meeting #4：start_time 有值，status: completed，持續約 3.5 分鐘
