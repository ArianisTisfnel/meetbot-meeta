## 二、角色定位

負責整個系統的「資料層 + API 層」。所有結構化資料、檔案儲存、向量索引的進出都經過你的程式碼。你提供的 API 是 Frontend（成員 B）唯一的後端入口；你維護的資料 Schema 是整個專案的「真相來源」。

### 核心職責

1. 設計與維護 Supabase Postgres 的 `app` schema
1. 開發 Hono Backend API（Materials CRUD、認證、權限）
1. 整合 Dify Knowledge Base API（上傳、查詢索引狀態、刪除）
1. 整合 Supabase Storage（檔案上傳、刪除）
1. 設計 rollback 邏輯，確保三個外部系統（Postgres、Storage、Dify）資料一致

### 不負責

- ❌ Vexa-lite 部署與 Bot 控制 → 成員 C
- ❌ 前端頁面開發 → 成員 B
- ❌ NextAuth 設定（你只負責驗 JWT，不負責簽 JWT）→ 成員 B

## 五、Phase 1 工作

### 5.1 Module 1：Supabase 與 Schema 分離設定（0.5 天）

**前置條件**：成員 C 已啟動 Vexa-lite，且 Vexa 已 migrate 完 `public` schema

**任務**：

```
□ 確認 Vexa public schema 完成（不要修改它）
□ 執行 SQL：CREATE SCHEMA IF NOT EXISTS app;
□ 設定 app schema 的權限：
    GRANT ALL ON SCHEMA app TO postgres;
    GRANT USAGE ON SCHEMA app TO authenticated;
□ 建立 Storage bucket: meeting-materials
  - 設定 private
  - File size limit: 15 MB
  - Allowed MIME types: application/pdf, application/vnd.openxmlformats-officedocument.wordprocessingml.document, text/plain, text/markdown
□ 取得 DATABASE_URL（連線字串尾要加 ?schema=app）
□ 取得 SUPABASE_SERVICE_ROLE_KEY
□ 用 prisma db pull 同步 public schema 的 Vexa 表定義
```

### 5.2 Module 2：Dify 環境正式建置（0.5 天）

```
□ 在 Dify 建立正式 Knowledge Base「meeting-materials」
□ 設定 embedding 模型：OpenAI text-embedding-3-small
□ 設定 retrieval：Hybrid search（語義 + 關鍵字）
□ 設定 chunking：chunk size 500 tokens、overlap 50 tokens
□ 取得 DIFY_API_KEY 與 DIFY_DATASET_ID
□ 用 curl 測試一次上傳 + 刪除流程
```

### 5.3 Module 3：Backend Hono API 開發（2 天）

#### 5.3.1 專案結構

```
backend/
├── prisma/
│   └── schema.prisma       # multiSchema 設定
├── src/
│   ├── index.ts            # Hono app entry
│   ├── env.ts              # 環境變數 schema (Zod)
│   ├── db.ts               # Prisma client
│   ├── lib/
│   │   ├── supabase.ts     # Supabase Storage client
│   │   ├── dify.ts         # Dify API client
│   │   └── hash.ts         # SHA-256 串流計算
│   ├── middleware/
│   │   ├── auth.ts         # JWT 驗證
│   │   └── require-role.ts # 角色檢查
│   ├── routes/
│   │   └── materials.ts    # /materials 相關 endpoints
│   └── jobs/
│       └── poll-indexing.ts # 定期同步 Dify 索引狀態
├── .env.example
├── package.json
└── tsconfig.json
```

#### 5.3.2 詳細任務

```
□ 專案初始化（Hono + TypeScript + Prisma + Zod）
□ 環境變數 schema 驗證（用 Zod 在啟動時檢查）
□ Prisma Schema 定義（詳見計劃書 7.3 節）
□ 跑 prisma migrate dev --name init_materials
□ Auth middleware：驗證 NextAuth JWT（用 jose library）
□ requireRole middleware
□ POST /materials
  - 接收 multipart/form-data（用 hono/multipart）
  - 驗證檔案類型、大小
  - 串流計算 SHA-256
  - 查重 (sha256 unique check) → 重複回 409
  - 上傳 Supabase Storage
  - 呼叫 Dify API 建立 document
  - Prisma create materials 紀錄
  - Rollback：任何步驟失敗都要清理已完成的步驟
□ GET /materials
  - User 角色：where uploadedByUserId = user.id
  - Admin 角色：where 1=1
□ GET /materials/:id（同步 Dify 索引狀態）
□ DELETE /materials/:id
  - 權限檢查：自己上傳或 admin
  - Dify 刪除 document
  - Supabase Storage 刪除檔案
  - Prisma update set deletedAt
□ Background job：每 30 秒 poll Dify 更新 processing 中項目
□ 統一錯誤格式：{ error_code, message, details }
□ Request logging（pino 或內建）
□ CORS 設定（允許 Dashboard origin）
```

### 5.4 Module 5 支援（整合測試，0.5 天）

```
□ 配合成員 B 解決 CORS 問題
□ 配合成員 B 驗證 JWT 流通
□ 提供測試帳號（Admin / User 各一）
□ 端對端測試清單（你的部分）：
```

## 十一、常見問題與注意事項

### Q1：Prisma 連線資料庫的 schema 該怎麼設定？

連線字串尾巴加 `?schema=app`，這樣 Prisma 預設操作的 schema 就是 `app`，不會誤觸 `public`。但跨 schema 查詢仍可用 `@@schema("public")` 在 model 上指定。

### Q2：Dify 索引是非同步的，使用者上傳後要等多久？

通常數秒到 1-2 分鐘，視文件大小而定。Backend 要透過 `GET /datasets/{id}/documents/{doc_id}/indexing-status` 查詢，Background job 每 30 秒 poll 一次更新本地狀態。

### Q3：Rollback 時若清理動作也失敗該怎麼辦？

至少要寫入日誌，並把該 material 標記為 `deletedAt` + `indexingStatus = FAILED`，留待人工或定期 cleanup job 處理。**永遠不要在 catch 中再 throw 新錯誤覆蓋原錯誤**。

### Q4：跨 schema 的 FK 為什麼不做？

因為 Vexa 的 `public.users` 表結構可能隨 Vexa 升級變動。建 FK 約束會讓你的 migration 卡住，且 Vexa 自己也不會替你維護一致性。**只用 ID 做邏輯關聯**，應用層自行處理孤兒資料即可。

### Q5：Supabase Service Role Key 可以放 Frontend 嗎？

**絕對不行**。Service Role Key 繞過所有 RLS，等同 admin。只能放 Backend env，前端絕對不要看到。
