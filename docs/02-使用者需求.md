## 一、認證與身份

### 1.1 登入

- 使用者透過 **Vexa Dashboard 的 NextAuth** 登入系統
- 支援的登入方式由環境變數決定，至少支援：
  - Google OAuth（首選）
  - Email Magic Link（需 SMTP 設定）
- 登入後系統在 Vexa `public.users` 建立或查找使用者紀錄，並發放 `vexa-token`
- 我們的 App（`app` schema）以 `public.users.id`（Integer，邏輯關聯，無 FK 約束）作為
  跨 schema 的使用者識別鍵，存為 `vexa_user_id`；email 僅作為 UX 層的邀請輸入方式
  （後端解析 email → vexa_user_id 後再儲存，不直接存 email）

### 1.2 身份

- 使用者在系統中的身份為**全域唯一**，不區分全域角色（Admin / User 等）
- 使用者對不同**專案**可有不同的角色（所有者 / 參與者），詳見第二節

## 二、專案（Project）管理

### 2.1 建立專案

- 使用者可在系統中建立一個或多個**專案實例**，並為其命名
- 建立者自動成為該專案的**所有者（Owner）**，此身份**永久不可移轉或撤銷**

### 2.2 角色與權限

專案內有兩種角色，各自擁有固定的權限組合：

| 角色 | 可執行的操作 |
|------|------------|
| **所有者** | 全部 5 種權限（檢視、編輯、刪除、管理、會議） |
| **參與者** | 檢視權 + 編輯權（共 2 種） |

#### 五種權限定義

| 權限 | 說明 |
|------|------|
| **檢視權** | 查看專案資料清單；查看唯讀的編輯歷史紀錄；查看所有者與參與者清單；查看進行中會議的逐字稿（唯讀） |
| **編輯權** | 上傳資料檔案至專案；刪除已上傳的資料檔案；每次上傳/刪除操作自動建立含「時間戳 + 操作者」的**唯讀歷史紀錄** |
| **刪除權** | 刪除整個**專案實例**（不可逆） |
| **管理權** | 賦予或移除其他使用者的「參與者」身分；調整參與者的「檢視權」或「編輯權」 |
| **會議權** | 在此專案下建立「會議實例」；邀請 Bot「小幫手」加入會議；讓「小幫手」離開會議；複製會議連結 |

### 2.3 邀請參與者

- 所有者在管理介面**輸入對方的 email** 進行邀請
- 系統查找該 email 是否對應到已登入過系統的 Vexa 使用者：
  - **存在**：直接加為參與者，預設賦予**檢視權**
  - **不存在**：顯示提示「對方尚未登入此系統，請對方先登入後再試」
- 所有者可隨時調整現有參與者的權限（檢視權 / 編輯權），或移除其參與者身分

## 三、資料（Materials）管理

### 3.1 檔案上傳流程

1. 進入該專案的檔案上傳介面
2. 支援三種上傳方式：
   - 點擊上傳按鈕，從本機選擇檔案
   - 從剪貼簿貼上檔案
   - 從其他視窗**拖拉**檔案至介面
3. 上傳的檔案先進入**暫存區**，使用者可：
   - 點擊「確認」（勾選）→ 正式上傳
   - 點擊「移除」→ 取消此次上傳
4. 確認後，檔案：
   - 儲存至 **Supabase Storage**（`meeting-materials` bucket）
   - 同步至 **Dify Knowledge Base** 作為 RAG 背景參考資料
5. 全程顯示即時狀態提示：
   - 「上傳中…」（含進度）
   - 「上傳完成」或「索引處理中」（Dify 索引為非同步）
   - 「索引完成」或「索引失敗」

### 3.2 支援的檔案格式與大小限制

| 項目 | 規格 |
|------|------|
| 支援格式 | PDF、DOCX、TXT、Markdown（`.md`） |
| 檔案大小上限 | 15 MB |
| 重複檔案 | 以 SHA-256 判重，重複檔案回傳提示 |

### 3.3 資料清單與管理

- 具備**檢視權**的成員可查看該專案所有已上傳的資料清單
- 清單顯示：檔名、上傳時間、上傳者、檔案大小、Dify 索引狀態
- 具備**編輯權**的成員可刪除資料檔案（三方同步刪除：Storage、Dify、資料庫）
- 刪除後寫入**唯讀歷史紀錄**（含時間戳與操作者）

## 四、會議實例（Meeting Instance）

### 4.1 建立會議實例

- 在頁面上方提供**下拉選單**，列出使用者「參與」的所有專案
- 選擇專案後，填入 Google Meet URL 以建立會議實例，有兩種輸入方式：
  - **手動輸入**：貼上已有的 Google Meet 連結
  - **快速建立**：點擊「建立新會議」→ 在新分頁開啟 `https://meet.new`，取得連結後貼回
- 會議實例有系統預設名稱（參考 Vexa Schema），建立後可自行命名

### 4.2 會議實例清單

- 一個專案可擁有**多個會議實例**，以清單方式呈現
- 清單顯示：會議名稱、建立時間、狀態（進行中 / 已結束）

### 4.3 不可刪除原則

- 會議實例**不可刪除**
- 理由：會議實例包含逐字稿等歷史資料，刪除會破壞協作紀錄與資料安全

### 4.4 會議實例檢視

點擊會議實例後進入其專屬介面，顯示以下資訊：

| 欄位 | 說明 |
|------|------|
| 會議名稱 | 可編輯（具會議權者） |
| 會議 URL | 顯示連結 + 一鍵複製按鈕 |
| 開始時間 | 自動記錄 |
| 結束時間 | 自動記錄（會議結束後顯示） |
| 與會人員 | 來自 Vexa 逐字稿中辨識的說話者 |
| 即時逐字稿 | 會議進行中持續更新（唯讀） |
| 會議摘要 | 會議結束後由 Dify 自動生成 |
| 交辦事項 | 會議結束後由 Dify 自動提取 |

## 五、Bot 互動（蜜塔 Meeta）

### 5.0 Bot 加入歡迎訊息

- Bot 成功加入 Google Meet 後，**自動**在聊天室發送一則歡迎訊息
- 歡迎訊息內容（以聊天室文字形式傳送，可視後續需求調整）：

```
嗨大家好！我是蜜塔（Meeta），你們今天的會議小幫手 🎉

你可以用語音或聊天室呼叫我：
  語音：說「蜜塔」或「小幫手」，再接上你的問題
  聊天室：輸入「蜜塔」或「小幫手」，再接上你的問題

我會根據此專案上傳的資料來回答問題，例如：
「蜜塔，請問去年 Q3 目標是什麼？」

有問題就找我！
```

- 歡迎訊息為**非同步**操作，不影響 Bot 加入的主流程回應

### 5.1 邀請與移除 Bot

- 具備**會議權**的成員可邀請「蜜塔（Meeta）」加入指定的 Google Meet
- 具備**會議權**的成員可讓「蜜塔（Meeta）」離開會議
- Bot 邀請使用**邀請者本人**的 Vexa API token 呼叫，不使用所有者帳號
  - 這樣每位使用者的 `max_concurrent_bots` 配額獨立計算，支援多個會議同時進行
  - 若邀請者已有 1 個進行中的 Bot（配額 = 1），系統顯示提示無法再建立

### 5.2 喚醒詞偵測

- Bot 持續監聽 Vexa Transcription API 的逐字稿 segments
- **喚醒詞**：「蜜塔（Meeta）」或「小幫手」（兩種皆可觸發）
  - 正規表達式：`/[蜜密祕秘迷][塔搭]|小幫手/`
  - 容錯設計：涵蓋輸入法常見錯別字（密塔、祕塔、秘塔、迷塔等）
- 系統有**防重複觸發**邏輯（例：同一段 segment 不觸發兩次）
- 偵測到喚醒詞後，擷取喚醒詞之外的文字作為**提問內容**

### 5.3 提問來源與回應方式

根據提問來源自動切換回應管道：

| 提問來源 | 系統提示（即時） | 回應方式 |
|---------|--------------|---------|
| **語音輸入** | 播放預錄音檔（「我收到了，正在查詢」） | Bot 以**語音**（OpenAI TTS）回覆答案 |
| **聊天室輸入** | 在 Meet 聊天室發送預存文字提示 | Bot 在**聊天室**以文字回覆答案 |

- 兩種管道皆不互相觸發（語音提問只語音回覆，聊天只文字回覆）
- 問題透過 **Dify Q&A 工作流**取得答案，無答案時有合理的 fallback 回覆

### 5.4 答案格式

- 語音回覆：Dify prompt 控制答案長度在 **100 字以內**（適合語音播報）
- 文字回覆：無強制長度限制，但建議簡潔

## 六、結束會議

### 6.1 結束觸發條件

- 退出 Google Meet（Vexa 偵測到 Bot 離開）即視為**結束會議**

### 6.2 會議結束後自動流程

1. 系統觸發 **Dify 總結工作流**
2. Dify 讀取該會議的**全量逐字稿**
3. 生成兩份輸出：
   - **會議摘要**：本次會議重點整理
   - **交辦事項**：從會議中提取的 Action Items
4. 結果自動更新至該**會議實例的檢視介面**

## 七、非功能性需求（附記）

| 類別 | 需求 |
|------|------|
| 安全 | `SUPABASE_SERVICE_ROLE_KEY` 僅存於後端，不暴露於前端 |
| 資料一致性 | 檔案上傳涉及 Storage、Dify、資料庫三方，任一步驟失敗需 Rollback |
| 狀態回饋 | 所有非同步操作（上傳、索引、摘要生成）須向使用者提供即時狀態提示 |
| 唯讀歷史 | 所有編輯操作（上傳 / 刪除檔案）自動建立含時間戳與操作者的唯讀歷史紀錄 |
| Bot 並發 | Vexa `public.users.max_concurrent_bots` 預設為 1，一位使用者同時只能有 1 個進行中的 Bot；Bot 邀請掛在邀請者帳號，不同使用者的配額互相獨立 |
| Bot 上下文隔離 | 每個進行中的會議在後端維護獨立的 session（各自的 WebSocket、防重複觸發狀態、isSpeaking 旗標），不同會議的對話不互相干擾；Dify Knowledge Base 刻意共用（同一專案的所有會議參考相同資料來源） |
