## 一、認證與身份

### 1.1 登入

- 使用者透過 **Vexa Dashboard 的 NextAuth** 登入系統
- 支援的登入方式由環境變數決定，至少支援：
  - Google OAuth（首選）
  - Email Magic Link（需 SMTP 設定）
- 登入後系統在 Vexa `public.users` 建立或查找使用者紀錄，並發放 `vexa-token`
- 我們的 App（`app` schema）以 `public.users.id`（Integer，邏輯關聯，無 FK 約束）作為
  跨 schema 的使用者識別鍵，存為 `vexa_user_id`
- **email 的儲存原則**：
  - **成員紀錄**（`project_members`）只存 `vexa_user_id`，不存 email
  - **邀請紀錄**（`project_invitations`）**會儲存 email**：受邀者可能尚未註冊，登入後必須靠
    「已驗證的 Google email」比對才找得到邀請，故存 email 為功能必需
    （此處覆蓋早期「後端不直接存 email」的敘述，詳見 03-資料庫Schema設計.md §4.7）

### 1.2 身份

- 使用者在系統中的身份為**全域唯一**，不區分全域角色（Admin / User 等）
- 使用者對不同**專案**可有不同的角色（所有者 / 參與者），詳見第二節

## 二、專案（Project）管理

### 2.1 建立專案

- 使用者可在系統中建立一個或多個**專案實例**，並為其命名
- 建立者自動成為該專案的**所有者（Owner）**，此身份**永久不可移轉或撤銷**

### 2.2 角色與權限

專案內有兩種角色，各自擁有固定的權限組合：

| 角色 | 可執行的操作 |
|------|------------|
| **所有者** | 全部 5 種權限（檢視、編輯、刪除、管理、會議），且所有權限**永久不可撤銷** |
| **參與者** | **檢視權為基準權限**（恆具備、不可取消）；可由所有者額外授予「編輯權」或「會議權」（各自獨立開關，見 §2.3） |

> **檢視權＝基準權限**：每位參與者一定具備檢視權，無法單獨關閉。編輯權、會議權是疊加在其上的加購開關。
> 若要收回某人的存取，所有者應**移除其參與者身分**，而非取消檢視權。

#### 五種權限定義

| 權限 | 說明 |
|------|------|
| **檢視權** | 查看專案資料清單；查看唯讀的編輯歷史紀錄；查看所有者與參與者清單；查看進行中會議的逐字稿（唯讀） |
| **編輯權** | 上傳資料檔案至專案；刪除已上傳的資料檔案；每次上傳/刪除操作自動建立含「時間戳 + 操作者」的**唯讀歷史紀錄** |
| **刪除權** | 刪除整個**專案實例**（不可逆） |
| **管理權** | 邀請或移除其他使用者的「參與者」身分；重寄/撤銷待處理邀請；調整參與者的「編輯權」或「會議權」（檢視權為基準權限，不可調整） |
| **會議權** | 在此專案下建立「會議實例」；邀請 Bot「小幫手」加入會議；讓「小幫手」離開會議；更新會議名稱；複製會議連結 |

### 2.3 邀請參與者（邀請制，可邀請尚未註冊者）

- 所有者在管理介面**輸入對方的 email** 並勾選初始權限（檢視權固定、可加選編輯權 / 會議權）進行邀請
- 系統**建立一筆「待處理邀請」並寄出邀請信**（不再要求對方必須先註冊）：
  - 對方**已有帳號**：登入後在左下角「📬 信箱」看到此邀請，可接受或拒絕
  - 對方**尚未註冊**：先收到邀請信，登入（Google OAuth）後系統以 email 比對，同樣在信箱看到邀請
  - 對方也可直接點邀請信中的連結，到 `/invitations/accept` 落地頁接受
  - **只有被邀請的那個 email 登入後**才能接受（系統強制比對 email，避免連結外洩被冒領）
- 邀請接受後對方才正式成為參與者，沿用邀請當下設定的權限
- 所有者在成員頁可看到「**邀請中（尚未接受）**」清單，並可**重寄**或**撤銷**未接受的邀請
- 所有者可隨時調整現有參與者的權限（編輯權 / 會議權；檢視權為基準權限不可取消），或移除其參與者身分
- **會議權的設計動機**：所有者可能因出差或其他原因無法出席主持，授予信任的參與者「會議權」後，對方即可自行邀請 Bot 並開啟會議，不需再等候所有者操作

> **寄信為盡力而為**：若後端未設定 SMTP，邀請仍會建立，但不寄信；此時介面會顯示一段可手動轉交的接受連結
> （連結僅顯示這一次）。詳見 06-後端架構.md `lib/email.ts`。

### 2.4 專案清單搜尋與篩選

- 專案總覽頁面提供：
  - **搜尋欄**：依專案名稱搜尋使用者所有相關專案（即時過濾）
  - **類型篩選**：「全部」（預設）、「我的」（使用者為 Owner 的專案）、「共用」（使用者為 Participant 的專案）
  - **排序**：「倒序」（預設，建立時間由新到舊）、「順序」（建立時間由舊到新）
- 以上三個篩選條件可組合使用

## 三、資料（Materials）管理

### 3.1 檔案上傳流程

1. 進入該專案的檔案上傳介面
2. 支援三種上傳方式：
   - 點擊上傳按鈕，從本機選擇檔案
   - 從剪貼簿貼上檔案
   - 從其他視窗**拖拉**檔案至介面
3. 上傳的檔案先進入**暫存區**，使用者可：
   - 點擊「確認」（勾選）→ 正式上傳
   - 點擊「移除」→ 取消此次上傳
4. 確認後，檔案：
   - 儲存至 **Supabase Storage**（`meeting-materials` bucket）
   - 同步至 **Dify Knowledge Base** 作為 RAG 背景參考資料
   - 若同時上傳多個檔案，後端需**依序（串行）**上傳至 Dify，每筆間隔約 1 秒，避免 Dify 回傳 502
5. 全程顯示即時狀態提示：
   - 「上傳中…」（含進度）
   - 「上傳完成」或「索引處理中」（Dify 索引為非同步）
   - 「索引完成」或「索引失敗」

### 3.2 支援的檔案格式與大小限制

| 項目 | 規格 |
|------|------|
| 支援格式 | PDF、DOCX、TXT、Markdown（`.md`） |
| 檔案大小上限 | 15 MB |
| 重複檔案 | 以 SHA-256 判重，重複檔案回傳提示 |

### 3.3 資料清單與管理

- 具備**檢視權**的成員可查看該專案所有已上傳的資料清單
- 清單顯示：檔名、上傳時間、上傳者、檔案大小、Dify 索引狀態
- 具備**編輯權**的成員可刪除資料檔案（三方同步刪除：Storage、Dify、資料庫）
- 刪除後寫入**唯讀歷史紀錄**（含時間戳與操作者）

## 四、會議實例（Meeting Instance）

### 4.1 建立會議實例

會議實例有兩個建立入口，行為相同：

#### 入口 A：在專案內建立
- 進入某個專案後，點擊「建立會議」按鈕
- 會議自動關聯此專案

#### 入口 B：全局建立（主導覽列）
- 在主導覽列（不需進入任何專案）點擊**「建立會議」**按鈕
- 開啟的對話框包含額外的**「關聯專案」**欄位：
  - 透過下拉選單或文字搜尋，列出使用者目前**擁有會議權**的所有專案
  - 此欄位**可留空**，建立無關聯專案的獨立會議實例（詳見 §4.2）

#### 共通欄位
- **會議名稱**（選填，預設由系統生成，格式：`"會議 YYYY-MM-DD HH:mm"`）
- **Google Meet URL**（必填），有兩種輸入方式：
  - **手動輸入**：貼上已有的 Google Meet 連結
  - **快速建立**：點擊「建立新 Meet」→ 在新分頁開啟 `https://meet.new`，取得連結後貼回

> **備注**：Google OAuth 是目前主要且優先的登入方式，可確保使用者具備 Google Workspace 存取權，便於開啟 `meet.new`。

### 4.2 全局 Meetings 頁面與無專案會議

- 主導覽列新增 **「Meetings」** 入口，顯示使用者近期參與的**所有會議實例**（跨專案整合視圖）
- 清單顯示：會議名稱、狀態（進行中 / 已結束）、關聯的專案名稱（若有）、開始/結束時間
- **無關聯專案的會議實例**：
  - Bot 仍可加入並提供即時逐字稿與會議摘要功能
  - **知識庫 Q&A 轉為逐字稿模式**：因無對應的 Dify Knowledge Base，Bot 收到喚醒詞後改以本次會議的**最近 30 段逐字稿**作為 context，透過 Claude（`claude-sonnet-4-6`）直接回答問題；歡迎訊息亦提示「查閱會議記錄」而非「查詢知識庫」
  - 無關聯專案的會議只出現在全局 Meetings 頁面，不出現於任何專案的會議清單中

### 4.3 會議實例清單

- 一個專案可擁有**多個會議實例**，以清單方式呈現於專案內的「會議」分頁
- 清單顯示：會議名稱、建立時間、狀態（進行中 / 已結束）

### 4.4 不可刪除原則

- 會議實例**不可刪除**
- 理由：會議實例包含逐字稿等歷史資料，刪除會破壞協作紀錄與資料安全

### 4.5 會議實例檢視

點擊會議實例後進入其專屬介面，顯示以下資訊（可從專案會議清單或全局 Meetings 頁面進入）：

| 欄位 | 說明 |
|------|------|
| 會議名稱 | 可編輯（具會議權者） |
| 會議 URL | 顯示連結 + 一鍵複製按鈕 |
| 開始時間 | 自動記錄 |
| 結束時間 | 自動記錄（會議結束後顯示） |
| 與會人員 | 來自 Vexa 逐字稿中辨識的說話者 |
| 即時逐字稿 | 會議進行中持續更新（唯讀） |
| 會議摘要 | 會議結束後由 Dify 會議摘要 Workflow 自動生成 |
| 交辦事項 | 會議結束後由 Dify 自動提取；每筆含**事項描述**與**負責人**（負責人可為空） |

### 4.6 會議清單搜尋與篩選

Meetings 全局頁面及專案內的會議清單均提供：
- **搜尋欄**：依會議名稱或關聯的專案名稱搜尋
- **時間篩選**：「全部」（預設）、「1 天內」、「3 天內」、「7 天內」（依會議建立時間或開始時間）
- **排序**：「倒序」（預設，由新到舊）、「順序」（由舊到新）

## 五、Bot 互動（蜜塔 Meeta）

### 5.0 Bot 加入歡迎訊息

- Bot 成功加入 Google Meet 後，**自動**在聊天室發送一則歡迎訊息
- 歡迎訊息內容（以聊天室文字形式傳送，可視後續需求調整）：

```
嗨大家好！我是蜜塔（Meeta），你們今天的會議小幫手 🎉

你可以用語音或聊天室呼叫我：
  語音：說「蜜塔」或「小幫手」，再接上你的問題
  聊天室：輸入「蜜塔」或「小幫手」，再接上你的問題

我會根據此專案上傳的資料來回答問題，例如：
「蜜塔，請問去年 Q3 目標是什麼？」

有問題就找我！
```

- 歡迎訊息為**非同步**操作，不影響 Bot 加入的主流程回應

> **⚠️ 商標限制**：基於 Vexa 的授權條款，前端介面不得展示 Vexa 的商標（名稱、Logo 等）。
> Bot 的對外顯示名稱統一使用「蜜塔（Meeta）」，內部技術術語（Vexa API、vexaToken 等）不在此限。
> 前端 Icon 設計應避免過度 AI 感的樣式，以符合品牌定位，必要時可移除圖示或改用中性風格。

### 5.1 邀請與移除 Bot

- 具備**會議權**的成員可邀請「蜜塔（Meeta）」加入指定的 Google Meet
- 具備**會議權**的成員可讓「蜜塔（Meeta）」離開會議
- Bot 邀請使用**邀請者本人**的 Vexa API token 呼叫，不使用所有者帳號
  - 這樣每位使用者的 `max_concurrent_bots` 配額獨立計算，支援多個會議同時進行
  - 若邀請者已有 1 個進行中的 Bot（配額 = 1），系統顯示提示無法再建立

### 5.2 喚醒詞偵測

- Bot 持續監聽 Vexa Transcription API 的逐字稿 segments
- **喚醒詞**：「蜜塔（Meeta）」或「小幫手」（兩種皆可觸發），並涵蓋英文辨識版本
  - 正規表達式：`/[蜜密祕秘迷][塔搭]|小幫手|[Mm]eeta|[Mm]ita/`
  - 容錯設計：涵蓋中文輸入法常見錯別字（密塔、祕塔、秘塔、迷塔等）
  - 英文版本：涵蓋語音辨識可能輸出的英文拼寫（Meeta、meeta、Mita、mita）
- 系統有**防重複觸發**邏輯（例：同一段 segment 不觸發兩次）
- 偵測到喚醒詞後，擷取喚醒詞之外的文字作為**提問內容**

### 5.3 提問來源與回應方式

根據提問來源自動切換回應管道：

| 提問來源 | 系統提示（即時） | 回應方式 |
|---------|--------------|---------|
| **語音輸入** | 播放預錄音檔（「我收到了，正在查詢」） | Bot 以**語音**（OpenAI TTS）回覆答案 |
| **聊天室輸入** | 在 Meet 聊天室發送預存文字提示 | Bot 在**聊天室**以文字回覆答案 |

- 兩種管道皆不互相觸發（語音提問只語音回覆，聊天只文字回覆）
- 問題透過 **Dify Q&A 工作流**取得答案，無答案時有合理的 fallback 回覆

### 5.4 答案格式

- 語音回覆：後端截斷確保答案不超過 **100 字**（適合語音播報；Chatflow Start 節點無 mode 輸入變數，截斷邏輯由後端 dispatchQuestion 實作）
- 文字回覆：無強制長度限制，但建議簡潔

## 六、結束會議

### 6.1 結束觸發條件

- 退出 Google Meet（Vexa 偵測到 Bot 離開）即視為**結束會議**

### 6.2 會議結束後自動流程

1. 系統取得該會議的**全量逐字稿**（含說話者與時間資訊）
2. 將逐字稿格式化為 **Markdown 檔案**並儲存至 Supabase Storage（永久保存）
3. 將 Markdown 檔案上傳至 **Dify 會議摘要 Workflow**（`POST /workflows/run`，以 file 方式傳入）
4. Dify 讀取檔案後生成三份輸出：
   - **會議摘要**：本次會議重點整理
   - **交辦事項**：從會議中提取的 Action Items（每筆含事項描述與負責人）
   - **關鍵議題 / 決議事項**：存入後端供未來查閱（目前 UI 以摘要為主要呈現）
5. 結果自動更新至該**會議實例的檢視介面**

## 七、非功能性需求（附記）

| 類別 | 需求 |
|------|------|
| 安全 | `SUPABASE_SERVICE_ROLE_KEY` 僅存於後端，不暴露於前端 |
| 資料一致性 | 檔案上傳涉及 Storage、Dify、資料庫三方，任一步驟失敗需 Rollback |
| 狀態回饋 | 所有非同步操作（上傳、索引、摘要生成）須向使用者提供即時狀態提示 |
| 唯讀歷史 | 所有編輯操作（上傳 / 刪除檔案）自動建立含時間戳與操作者的唯讀歷史紀錄 |
| Bot 並發 | Vexa `public.users.max_concurrent_bots` 預設為 1，一位使用者同時只能有 1 個進行中的 Bot；Bot 邀請掛在邀請者帳號，不同使用者的配額互相獨立 |
| Bot 上下文隔離 | 每個進行中的會議在後端維護獨立的 session（各自的 WebSocket、防重複觸發狀態、isSpeaking 旗標），不同會議的對話不互相干擾；Dify Knowledge Base 刻意共用（同一專案的所有會議參考相同資料來源） |
