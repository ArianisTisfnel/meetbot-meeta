## 一、背景與方法

上一輪「**專案成員邀請**」功能（pending 邀請＋email 寄送、`canView` 改為基準權限、新增兩張表與相關端點、Dify dataset 預設權限改 `all_team_members`）落地後，`docs/02–06` 多處未回寫，文件落後於程式。

本次逐份比對 `backend/prisma/schema.prisma`、`backend/src/{routes,services,sessions,lib,middleware,types}`、`frontend/src/{app,components,hooks}`，列出落差表（類型 ①已實作未回寫／②文件規格未實作／③文件與程式矛盾），並依「**以程式為準、回寫文件**」原則修正。

**結論**：落差幾乎全為 ①／③，且方向單一（文件落後於程式），故選擇一次補齊的「修正報告」。唯一程式側調整為刪除一個重複的 dead function。

## 四、授權缺口修正（已處理，採 (a)）

掃描時發現一個**後端授權未對齊設計意圖**的缺口（屬程式 bug）：

- **`POST /projects/:id/meetings/:mid/bot/leave` 與 `.../cancel`** 的 route 原本只透過 `getProjectMeeting` 驗證**檢視權**，
  未在後端再強制 `canMeeting`——只有檢視/編輯權的參與者**直接打 API 即可結束或取消會議**（前端按鈕雖隱藏，API 可繞過）。
- 對照：`bot/reinvite` 已在 `reinviteBot` service 內部驗證「建立者 / owner / canMeeting」，是正確的。

**處置（採 (a) 改程式對齊文件）**：
- 新增 `meeting.service.requireProjectMeetingManageAccess(projectId, meetingId, vexaUserId)`：驗證 Owner 或 `canMeeting`，
  並確認會議屬於該專案；在 `bot/leave`、`cancel` 的專案版 route 改呼叫它（取代原本只驗檢視權的 `getProjectMeeting`）。
- 未改 `leaveMeeting` / `cancelMeeting` 本身（兩者亦由全局路由以 `createdBy` 授權呼叫，權限在呼叫端把關）。
- 文件已對齊：04 §九 授權備註、§十 權限矩陣（移除 ³ 待修註，bot/leave、cancel 列回 `canMeeting`）。
- 影響：行為變更——僅檢視/編輯權成員不再能透過 API 結束/取消會議（符合設計意圖）。
